  1. _At ~5 agents, GPU throughput stays flat, p99 latency explodes due to serialized handoffs, context merges, and router queueing. Compute isn’t saturated, coordination depth is._
  2. _A few hundred KB per agent silently multiplies into MB-scale payloads per request. Tokens, memory, and latency balloon superlinearly even when models are fast._
  3. _Past 5+ agents, ownership diffuses. Agents optimize for safe micro-changes, abandon hard cases, and recreate human meeting dynamics in silicon._
  4. _Persistent context causes drift, reconvergence loops, and variance inflation well before context windows fill. Failures are probabilistic, not catastrophic and nearly impossible to debug._
  5. _Stateless, isolated agents with strict I/O contracts + event-sourced planners deliver deterministic outcomes. Intelligence lives in orchestration, not agents. Simplicity is an economic constraint, not a style choice._


This wasn’t a moonshot system. It was a very normal, very pragmatic real-time fraud detection pipeline, handling roughly $25M a day in card and ACH traffic. The mandate was simple: low latency, explainable decisions, clean audit trails. We built the system using Langchain & LangGraph for explicit orchestration and state transitions, with OpenAI’s GPT-OSS 20B as the underlying model. We started with four narrowly scoped agents, each deliberately constrained. One handled feature enrichment from transaction context. One focused purely on behavioral scoring. One applied policy and risk rules. The final agent assembled the explanation and audit record with recon.
At four agents, the system felt almost boring in the best way. Median latency sat around 380–420ms, and the 99th percentile hovered just under 800ms, which was exactly where the SLA needed it to be. Nothing clever. Nothing heroic. The demo looked clean. The alerts made sense. Ops was happy. Leadership approved the rollout without much debate.
Not catastrophically at first. Just enough that engineers began adding “small” fixes. More context passed between agents. A bit more shared state to reduce duplication. One agent started peeking at another’s intermediate output to “improve accuracy.” None of this felt unreasonable in isolation. In fact, it felt responsible.
But by the time we re-ran a full load test, the system no longer behaved like the one we had shipped.
The 99th percentile jumped into the multi-second range, blowing past the SLA hard enough that retries kicked in and made things worse. What caught us off guard was that model throughput still looked fine. Tokens per second were healthy. GPU utilization wasn’t pegged. On paper, nothing was “slow.”
Once we instrumented the handoffs properly, the pattern was impossible to ignore. Each agent was carrying a growing slice of what it believed was “necessary context” transaction metadata, partial risk assessments, explanations-in-progress, confidence scores. On average, each agent was hauling around a couple of megabytes of serialized state per decision. Every time an agent updated its view, it pushed that state downstream.
With four agents, that already meant multiple synchronization points per transaction. Not dozens just enough to matter. Router queues started backing up after 10–15 concurrent handoffs, not because the models were overloaded, but because coordination had become serialized. Each agent was now waiting on confirmations from the others before proceeding.
Nearly half of our token spend wasn’t going toward fraud detection logic anymore. It was going toward agents restating, reconciling, and justifying intermediate conclusions to each other. We had effectively turned a low-latency decision pipeline into a distributed consensus system.
Every agent was well-intentioned. Every piece of context felt defensible. But the system as a whole had crossed a threshold where coordination cost dominated useful work. The failure wasn’t dramatic in the logs. It was structural. And once you cross that line, no amount of micro-optimization brings you back.
That was the point where it became clear we were hitting an architectural limit. Adding another agent didn’t increase capacity; it increased serialization. From then on, coordination stopped being something we “optimized later” and became the first constraint we designed around.
The part nobody tells is things don’t break at a 50+ agents. They start bending much earlier. For us, the inflection point showed up just past five.
Once we crossed roughly five to eight agents in a flat topology, the system didn’t just slow down it changed character. Decisions became softer. Outputs became cautious. Instead of committing to a strong hypothesis or taking ownership of a hard call, agents started optimizing for local safety. We saw a steady drift toward micro-edits: slight threshold adjustments, narrowly scoped rule refinements, tiny confidence tweaks that were technically correct but strategically irrelevant.
The hard cases the ambiguous fraud patterns, the edge behaviors that actually mattered were increasingly left untouched. No agent wanted to be the one that overreached. Responsibility diffused. The behavior was uncomfortably familiar, because it looked exactly like what happens in human teams without clear ownership and hierarchy. Agent capacity increased, but effective throughput collapsed under coordination cost.
What made this especially frustrating was that nothing was “wrong” at the component level. Each agent was performing well in isolation. Accuracy metrics looked fine. Unit tests passed. But system-level progress stalled. 
Under the hood, the failure mode was brutally simple. As the agent count grew, every agent’s output implicitly became part of every other agent’s reasoning context. Even without explicit messaging, the shared assumptions, partial summaries, and evolving state created serial dependencies everywhere. Coordination costs didn’t grow linearly; they ballooned. The effective overhead scaled like O(n²), not in FLOPs, but in tokens consumed, latency added, and cognitive load imposed on the system. Each additional agent increased serial dependencies and coordination cost, reducing net system capacity.
The reason this catches teams off guard is that none of it shows up in early experiments. With two or three agents, the coordination tax is invisible. Even at five, it feels manageable. It doesn’t surface in unit tests, benchmark harnesses, clean demos. You only see it when the system is under real load, handling real variance, with no human in the loop to smooth over the cracks.
When I later saw the Google/MIT result showing that once single-agent accuracy passes ~45%, adding agents hurts system performance, it wasn’t a surprise. We’d already seen it in production. By agent 10, the system was clearly worse than it had been at 5 despite having more intelligence on paper.
At the time, the industry consensus felt settled almost boring in how obvious it sounded. Agents should be autonomous. They should collaborate like human teams. They should run continuously, accumulating context so they learn over time. Shared state, we were told, would keep them aligned and reduce duplication. Every framework, demo, and conference talk reinforced the same mental model.
At that scale, low fan-out and small state hide coordination cost. Latency looks acceptable, memory stays stable, and systemic failure modes don’t surface yet.
We ran agents as long-lived processes, each carrying its own evolving internal context. Over time, that context stopped being an asset and started behaving like technical debt. Even before we hit context window limits, performance degraded. Earlier high-signal facts were still present, but they were diluted by layers of intermediate reasoning, partial conclusions, and stale assumptions. The model wasn’t forgetting so much as losing its ability to discriminate what still mattered. We started calling it context pollution not because the data was wrong, but because relevance decayed faster than we could prune it.
Shared state made things worse in a way that only shows up under failure. We had agents reading and writing overlapping artifacts intermediate plans, partial scores, confidence annotations, assuming they’d converge. Most of the time they did. But when an agent crashed mid-write or retried with slightly different inputs, downstream agents consumed half-formed state and treated it as ground truth. Idempotence vanished. Replays didn’t replay. A restart didn’t restore the system; it forked it.
Debugging stopped being a matter of tracing a request. It became an exercise in reconstructing causal history across multiple agents, each with its own evolving view of the world. Logs were technically complete and practically useless. We weren’t debugging code anymore we were debugging emergent behavior.
The deeper issue was that we had quietly imported human coordination patterns into a system that doesn’t tolerate them. Meetings, shared documents, long-running threads of reasoning these are already inefficient for people, but people compensate with intuition and social repair mechanisms. Agents don’t. When you give them shared state and continuous operation, you create serial dependencies on their own past decisions and on each other’s partial outputs. Latency increases, retries multiply, and small inconsistencies amplify instead of canceling out.
3 months in, the failure was clear. We didn’t scale intelligence; we scaled coordination cost. Each added agent increased contention, inconsistency, and drift. The system didn’t improve it destabilized.
What finally broke the spell was realizing that none of these failures showed up in unit tests, demos, or early prototypes. They only appeared once the system was stressed enough that coordination became the dominant cost. The industry consensus wasn’t wrong it was incomplete. It described what works at 5 agents and quietly fell apart at 10–50.
By the time we accepted that, we’d lost three months. But we’d also learned the lesson that mattered: scale doesn’t punish bad models first. It punishes bad assumptions about coordination.
Nothing improved until we stopped optimizing agents and started treating coordination as the dominant cost. Past a small N, coordination overhead- not inference, defined latency, throughput, and failure modes.
We scrapped the agent team idea entirely and rebuilt the system as a strict two-tier control plane. At the top sat a planner and a judge. The planner’s only job was to take an incoming request and deterministically decompose it into independent units of work no reasoning about execution, no optimization, just clean task boundaries. The judge operated downstream, validating outputs against explicit invariants: schema correctness, confidence thresholds, policy constraints, and cross-task consistency checks.
The worker tier was intentionally dull. Hundreds at first, then thousands. Each worker received a narrowly scoped task with tightly bounded input: usually a single document, a single feature slice, or a single hypothesis to validate. The context window was aggressively trimmed often under 3% of what our earlier agents carried. No global state, no shared memory, no awareness of what other workers were doing or why the task existed in the first place.
A worker executed once, produced a concrete artifact — JSON, protobuf, diff, scorecard persisted it to an external store, and terminated. No warm state. No accumulating context. No learning across invocations. When it felt wasteful, we reminded ourselves that statelessness was the feature, not the bug.
This design felt almost heretical at first. We’d spent so much time trying to make agents more capable, more informed, more autonomous. Now we were doing the opposite deliberately constraining them, stripping away context until only the minimum viable signal remained. It felt like we were hobbling intelligence.
The impact was immediate and measurable. End to end latency dropped by 73% in the first full production run. More importantly, the long tail collapsed. The p99 stopped wandering because execution paths were no longer coupled through shared context or implicit coordination. Throughput scaled linearly with worker count until we hit actual infrastructure limits, not architectural ones.
Debugging changed overnight. When something failed, we didn’t have to reconstruct an agent’s mental state across ten prior turns. We just replayed the task with the same input and got the same output. If a worker crashed, it didn’t matter, we reissued the task. Progress lived in the orchestration layer, not in any agent’s head.
That’s when the system became idempotent at the workflow level. Individual executions were stochastic workers timed out, retried, or failed but progress was preserved because all state lived outside the agents. Every transition was versioned, replayable, and recoverable. Errors no longer caused drift; they simply triggered re-execution from a known checkpoint.
In hindsight, the lesson was obvious. We stopped trying to build a society of intelligent agents and started building a distributed system with LLMs as pure compute primitives. Once we drew that line, scale stopped being mysterious — and started behaving like engineering again.
The shift that finally stabilized the system wasn’t a new model or a clever routing trick. It was much more mundane and much more powerful. We stopped writing prompts like instructions to a smart assistant and started treating them like API contracts between distributed systems.
Before that change, our prompts were expressive but vague. They explained intent, gave examples, encouraged reasoning. They also quietly embedded assumptions about what other agents would do, what context would be available, and how failures should be interpreted. At small scale, that ambiguity was survivable. At 5, 10, 15 agents, it showed up as chaos not in obvious crashes, but in slow, compounding drift.
We replaced it with something much stricter. Every worker ran against a fixed contract: typed inputs, bounded context, and a machine-validated output schema. Most inputs were serialized objects, not free text. Outputs had to pass validation or fail fast. If confidence dropped, fields were missing, or ambiguity crossed a threshold, the agent returned a structured error and stopped.
Before, multi-agent failures surfaced as diffuse symptoms: rising tail latency, oscillating confidence scores, inexplicable retries, or downstream agents producing logically consistent but semantically incompatible outputs. When we instrumented traces, the exception graphs looked like spiderwebs errors propagating sideways across agents because nobody knew where responsibility actually ended.
After the shift, failures became local and legible. Validation errors spiked early and sharply, then flattened. Trace graphs collapsed from tangled meshes into clean trees. We could point to a single contract violation and say, “This agent broke its promise,” instead of reverse-engineering intent from logs. Evaluation became continuous and cheap: every agent output was automatically checked against its contract, and eval scores were tracked the same way we tracked latency or error rates.
Nearly 80% of our multi-agent failures vanished once we eliminated interface ambiguity. The models hadn’t improved; the contracts had. What we’d been chasing as reasoning errors were, in practice, spec violations caused by poorly defined boundaries. We weren’t debugging intelligence we were debugging interfaces.
MCP-style context separation helped reinforce this discipline. System context defined global rules and guardrails and almost never changed. Operational context : task state, intermediate artifacts, checkpoints lived outside the agent and was versioned and observable. Showing up in a prompt without a contractually defined reason became a code smell. Context sharing stopped being implicit and became intentional.
Once prompts behaved like APIs, coordination infrastructure started to evaporate. We didn’t need agents negotiating with each other because the contracts already encoded who was responsible for what. We didn’t need long-lived memory to stay aligned because alignment was enforced at the boundary, not learned implicitly over time.
The result wasn’t elegant in a research-paper sense. It was boring. Predictable. Replaceable. And that was the point. The system scaled not because the agents were clever, but because they were constrained — and because every failure left behind a clean trail in telemetry instead of a philosophical debate about what the model “meant to do.”
Up to that point, everything looked healthy. Single-agent accuracy was solid. End-to-end flows passed integration tests. Even average latency was fine. But when we turned on real traffic bursty, skewed, and very Zipfian the behavior changed immediately. A small fraction of requests triggered deep reasoning paths, heavy tool usage, and large intermediate artifacts. That’s where things started to bend.
Each agent only needed a few hundred kilobytes of intermediate state — partial reasoning, tool outputs, embeddings, structured traces. That sounded harmless until those artifacts were serialized, passed through the orchestration layer, embedded into prompts, and rehydrated by downstream agents. With just 5–7 agents, fan-out multiplied that footprint into megabytes per request. Caching didn’t help because the state wasn’t reusable. The access pattern was path-dependent. What finally stabilized memory was forcing agents to be stateless and pushing all intermediate state into external stores: Git-like object storage and task queues so agents only handled references, never payloads.
We started seeing latency spikes that had nothing to do with model inference. GPUs were underutilized. CPU wasn’t pegged. But request traces showed growing gaps between agent invocations. When we plotted queue depth and concurrency, it became obvious: every LLM router we tested saturated at roughly 20 concurrent handoffs. Not requests handoffs. Each handoff was a serialized dependency where one agent’s output gated the next step. No amount of horizontal scaling fixed it because this wasn’t a throughput problem. It was queuing theory. You can’t parallelize serial dependencies.
This showed up clearly in latency histograms. p50 stayed reasonable. p99 exploded as soon as coordination depth increased. Every additional agent added another tail-latency multiplier. At that point it was clear this wasn’t a vendor issue or a tuning problem. It was physics.
Around 10 agents, cost curves flipped. Cost per thousand predictions climbed to $2.14 against a hard budget of $0.43. The spend wasn’t coming from inference complexity it was coming from coordination. Token accounting showed that a huge portion of tokens were being consumed by agents summarizing state, reconciling decisions, and justifying outputs to other agents. Training masked this because coordination cost is paid once. Inference pays it on every request.
Hardware limits amplified everything. GPU memory fragmentation increased as context windows grew. Vector serialization between agents added milliseconds per hop that compounded under load. Even shared feature stores started showing I/O contention once concurrency crossed single digits. None of this was theoretical — we saw it in profiling traces, queue depth metrics, and retry showing up in places retries should never exist.
Observability is the only reason we figured this out. We didn’t guess. We had per-agent token throughput, hop-by-hop latency histograms, queue saturation alerts, and retry-rate dashboards. We watched retry storms form when downstream agents timed out and upstream planners retried aggressively without backpressure. The orchestration layer had to learn restraint hard fan-out limits, backpressure, priority queues, and retry suppression to stay upright.
We stripped shared state. We constrained context aggressively. We forced agents to be stateless and short-lived. All global reasoning moved into the orchestration layer. The control plane tracked progress, enforced budgets, handled retries, and absorbed failures. Agents became disposable compute units: restartable, reproducible, and deliberately ignorant.
Once we did that, scaling stopped being scary. Latency curves flattened. Cost became predictable. Failure modes stopped multiplying. Debugging felt like engineering again instead of folklore.
The takeaway is simple: multi-agent systems don’t break because the ideas are wrong. They break because queues back up, memory explodes, networks serialize, and coordination leaks into places you didn’t budget for.
Here’s my blunt take after actually living through 3 → 5 → 10 agents in tier-1 production — where “scale” isn’t a demo metric, it’s a power bill, a latency SLO, and a pager at 2 a.m.
The first wall we hit wasn’t model quality. It was physics. Once we crossed roughly five concurrent agents per request, GPU utilization looked healthy on paper — 70–75% steady — but tail latency told a different story. Token throughput flattened while end-to-end latency crept past 1.6s, then 2.4s, even though per-agent inference stayed constant. The issue wasn’t FLOPs; it was coordination. Each additional agent introduced another serialization point — another round-trip over the network, another context merge, another cache miss. We saw GPU kernels idle while CPUs waited on orchestration locks. In a few cases, thermal throttling kicked in simply because we were stretching inference windows long enough to sustain peak power draw — 450–500W per device — far longer than the system was designed for.
Long-running autonomy made this worse, not better. Agents that persisted across tasks accumulated context that felt useful initially, but over time turned into a liability. Memory usage oscillated, vector stores churned, and reasoning quality drifted in subtle ways — agents began reconverging on earlier hypotheses even when fresh evidence contradicted them. Nothing crashed outright. Instead, we saw what I can only describe as temporal mush: slower decisions, repetitive reasoning loops, and rising variance in outputs. Debugging was brutal because the failure wasn’t a bug — it was entropy.
Shared state was the quiet killer. We tried every variation: shared tool access, shared scratchpads, even shared embeddings. At small scale it felt elegant. At ten agents, it became contention. We saw retries stack up, MCP-style orchestration queues back up under load, and network chatter spike just enough to push us over our latency ceiling. Once agents could observe each other indirectly through shared state, they began compensating for one another — double-checking, hedging, deferring. It looked like collaboration. It was actually paralysis.
The turnaround came when we stopped treating agents like teammates and started treating them like stateless execution cells. Each worker ran in its own containerized reasoning boundary with a strict input/output contract. No shared memory. No awareness of siblings. Communication happened through an event-sourced planner over a message bus — Kafka in our case — where the planner owned task decomposition, retries, and validation. Workers were ephemeral: spin up, reason, emit an artifact, terminate. Memory lived outside the agent, indexed and versioned. When something failed, we didn’t “recover” the agent — we replayed the event.
From up close, this system looked dumb. The workers had no global intelligence. But the orchestration layer became deeply sophisticated: adaptive fan-out based on load, backpressure-aware scheduling, latency budgets enforced per hop. Once we showed the scheduler token cost per task and watt-seconds per decision, design choices became obvious. Simplicity wasn’t aesthetic it was economic. We finally converted compute into capability instead of burning it on coordination overhead.
Looking ahead, the real gains won’t come from more capable agents, but from better orchestration: schedulers that account for GPU power envelopes and thermal limits, planners that dynamically cap fan-out based on live latency budgets, and memory layers that are vector-indexed, versioned, and aggressively scoped rather than accumulated. The architectures that win won’t look autonomous they’ll look controlled.
We only learned that after the system broke at scale, not during design reviews. If this saves you a few months of iteration and a material amount of token spend it did its job.
